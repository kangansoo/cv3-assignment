# AGENTS.md

## 프로젝트 규칙

이 프로젝트는 TypeScript 기반의 작은 Next.js 애플리케이션입니다.

코드는 단순하고 읽기 쉬우며, 직접 설명할 수 있는 형태를 우선합니다. 현재 요구사항을 벗어나는 기능, 추상화, 의존성은 추가하지 않습니다.

## 개발 규칙

- 코드 구현은 사용자의 승인을 받은 후에만 진행합니다.
- 애플리케이션 코드는 TypeScript로 작성합니다.
- 복잡한 추상화보다 단순한 React 컴포넌트를 선호합니다.
- 데이터 요청 및 정규화 로직은 UI 컴포넌트와 분리합니다.
- 명확히 필요하지 않다면 전역 상태 관리 라이브러리를 추가하지 않습니다.
- 이미 설치되어 있지 않은 스타일링 라이브러리나 UI 라이브러리를 추가하지 않습니다.
- 테이블 렌더링 로직은 예측 가능하고 추적하기 쉽게 유지합니다.
- 화면에 표시되는 데이터 값은 원본 데이터와 최대한 동일하게 유지합니다.
- 관련 없는 파일이나 코드 포맷은 변경하지 않습니다.

## 문서 관리 규칙

새로운 작업을 시작하기 전에 `docs/work.md`에 작업 계획을 작성합니다.

`docs/work.md`에는 다음 내용을 간단히 남깁니다.

- 작업 목표
- 구현 범위
- 예상 수정 파일
- 검증 방법

작업을 완료한 뒤에는 `docs/history.md`에 작업 결과를 작성합니다.

`docs/history.md`에는 다음 내용을 기록합니다.

- 작업 날짜
- 변경 내용
- 검증 결과
- 남은 이슈 또는 후속 작업

작업 기록은 길게 작성하지 않고, 나중에 작업 흐름을 이해할 수 있을 정도로만 간결하게 유지합니다.

## 파일 구조

- 재사용 가능한 UI 컴포넌트는 `src/components`에 둡니다.
- 데이터 요청 및 변환 로직은 `src/lib`에 둡니다.
- TypeScript 타입은 `src/types`에 둡니다.
- 서버 측 데이터 접근은 `src/app/api` 아래의 Next.js Route Handler를 사용합니다.

## 코드 스타일

- 짧은 이름보다 의미가 분명한 이름을 사용합니다.
- 함수는 작고 하나의 역할만 하도록 작성합니다.
- 가독성이 좋아지는 경우 early return을 사용합니다.
- 주석은 코드의 의도가 명확하지 않은 경우에만 작성합니다.
- 사용하지 않는 변수나 함수, 불필요한 주석은 commit하지 않습니다.

## 주석 작성 규칙

제출용 코드이므로 필요한 곳에는 주석을 작성합니다.
다만 모든 함수, 변수, 로직에 기계적으로 주석을 달지 않습니다. 함수명과 변수명만으로 의도가 분명한 코드는 주석을 생략합니다.
주석은 코드가 "무엇을 하는지"보다 "왜 이렇게 처리하는지"를 설명하는 데 사용합니다.

다음과 같은 경우에만 주석을 작성합니다.

- 원본 데이터 구조가 화면 표시 형식과 달라 변환이 필요한 경우
- 특정 필드를 원본 문자열 그대로 유지해야 하는 경우
- 예외 처리나 fallback 값이 필요한 이유가 있는 경우
- 서버 컴포넌트, 클라이언트 컴포넌트, API Route의 역할 구분을 설명할 필요가 있는 경우
- 면접에서 설명할 만한 핵심 의사결정이 코드에 반영된 경우

다음과 같은 주석은 작성하지 않습니다.

- 코드 내용을 그대로 반복하는 주석
- 변수명이나 함수명으로 이미 알 수 있는 설명
- 오래된 구현 흔적이나 TODO성 메모
- 임시 디버깅 설명

## 커밋 규칙

- 커밋 메세지는 한국어로 작성합니다.
- 작업 내용은 하나의 커밋으로 뭉치지 않고, 의미 있는 작업 단위로 나누어 커밋합니다.
- 각 커밋은 가능한 한 하나의 목적만 가지도록 작성합니다.
- 커밋 전에는 변경 파일을 확인하고, 현재 작업과 관련 없는 변경 사항이 포함되지 않도록 합니다.

```bash
git status
git diff
```

- 커밋 메시지는 다음 형식을 사용합니다.
  - chore
  - feat
  - docs
  - style
  - fix
  - refact

- 커밋 메세지 예시

```
feat: 방송 유형 전환 기능 구현
```
